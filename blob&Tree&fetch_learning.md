# 📚 Git 底层原理与高阶命令完整学习笔记

## 💡 核心思维导图：Git 的“静”与“动”
Git 的本质并不是记录差异，而是一个基于**键值对（Key-Value）的哈希文件系统**。我们在项目里的所有操作，都可以拆解为“静”与“动”两个维度：

* **【静：底层对象】**：`Blob` (纯数据内容) ──> `Tree` (目录结构与文件名) ──> `Commit` (版本快照与时间线)
* **【动：数据流动】**：`Git Fetch` (安全同步远端) ──> `Git Rebase / Merge` (推进与整合历史)

---

## 第一部分：Git 三大核心底层对象（静止的快照）



### 1. 数据层：Blob（二进制大对象）
* **核心定义**：Blob（Binary Large Object）是 Git 存储**文件内容**的最小基本单位。
* **关键特性**：
    * **只管内容，不管名字**：Blob 内部完全剥离了文件名、路径和修改时间等元数据。它只纯粹地保存文件里的代码或文本。
    * **天然去重机制**：如果项目中有两个完全相同的文件（即使它们在不同的文件夹、叫不同的名字），Git 在后台**只会生成一个 Blob 对象**，极大地节省了存储空间。

### 2. 结构层：Tree（树对象）
* **核心定义**：如果说 Blob 是“文件内容”，那么 Tree 就是**“文件夹（目录）”**。它负责把散落的 Blob 重新组织成人类能看懂的项目结构。
* **关键作用**：
    1.  **映射文件名**：将真实的文件名/文件夹名与底层的 SHA-1 哈希值绑定。
    2.  **构建目录树**：Tree 对象既可以指向 Blob（代表文件），也可以指向另一个 Tree（代表子文件夹），从而递归形成一棵完整的项目目录树。
    3.  **记录文件权限**：记录系统级别的文件模式（如 `100644` 代表普通文件，`040000` 代表目录，`100755` 代表可执行文件）。
* **解剖 Tree 的内部结构**：
    ```text
    # 权限模式   对象类型   哈希值 (40位)                   文件名/文件夹名
    100644      blob      8a30a28b68aa644a696a1a...      README.md
    040000      tree      3b4c5d6e7f8a9b0c1d2e3f...      src
    ```

### 3. 历史层：Commit（提交对象）
* **核心定义**：代码在特定时间点的完整“快照记录”。
* **关键作用**：每一个 Commit 都会指向项目根目录那一刻的**顶层 Tree 对象**，并附带作者信息、时间戳、提交说明（Commit Message）以及**父提交（Parent Commit）的哈希值**。正是由于父提交的存在，才串联出了 Git 的历史时间线。

## 第二部分：底层对象的存储与寻址机制

无论对象是 Blob、Tree 还是 Commit，Git 在后台处理它们的流水线是完全一致的：

> **存储流水线**：
> 文件数据/列表文本 ──> 拼接头部信息 ──> 计算 SHA-1 哈希值 (40位) ──> `zlib` 压缩 ──> 写入磁盘

* **物理存储位置**：所有的底层对象，都被压缩保存在项目根目录的 `.git/objects/` 文件夹下。
* **分段目录设计（性能优化）**：为了防止单目录下文件过多导致系统检索变慢，Git 会截取 40 位哈希值的**前 2 位作为文件夹名**，剩下的 **38 位作为文件名**。
    > *示例：一个哈希值为 `7a1b2c3d4e...` 的对象*
    > *实际存储路径为：`.git/objects/7a/1b2c3d4e...`*

## 第三部分：核心动态命令（数据的同步与修剪）



### 1. 安全的下载器：Git Fetch
* **核心定义**：`git fetch` 是一个**绝对安全、只读**的同步命令。用于去远程仓库（如 GitHub）抓取你本地没有的最新提交、分支和标签。
* **工作机制**：
    * 它只负责把数据下载到本地的 `.git` 目录中。
    * 它会更新你本地的**远程追踪指针**（如 `origin/main`），但**绝不会自动合并**到你当前正在开发的分支，也不会修改你的工作区代码。
* **黄金公式**：
    $$\text{git pull} = \text{git fetch} + \text{git merge}$$
    * **建议工作流**：先执行 `git fetch`，使用 `git log main..origin/main` 观察远程有哪些新变动，确认无误后再决定用 merge 或 rebase 整合代码。



### 2. 历史修剪术：Git Rebase（变基）
* **核心定义**：强行改变当前分支的“基底”（起点）。将当前分支的提交，像“移花接木”一样，按顺序重新应用到目标分支的最顶端。
* **核心优势**：可以消灭因为 `merge` 产生的交错分叉，让团队的整体提交历史变成一条干净、清晰的**直线**。
* **变基三步曲（以 `git rebase main` 为例）**：
    1.  **暂存**：将当前开发分支（如 `feature`）上特有的提交摘取下来，放入临时区域。
    2.  **重置起点**：将当前分支的底层指针强行指向 `main` 分支的最新提交。
    3.  **重放**：把刚刚暂存的提交，按顺序依次重新应用到新的起点之后。*(⚠️ 注意：这个重新应用的过程会生成全新的哈希值，意味着它们已经变成了全新的提交)*

### ⚠️ Rebase 的绝对铁律（黄金法则）
> **永远不要在已经推送到远程共享仓库的公共分支上使用 `git rebase`！**

* **原因**：Rebase 会改写历史记录（改变哈希值）。如果对已经公开的分支强行变基并推送到远端（force push），会导致其他同事本地的提交基底瞬间断裂，引发灾难性的代码冲突。
* **正确用法**：只在你**本地尚未推送**的私有分支上使用 rebase，用来整理自己凌乱的提交记录（如使用 `git rebase -i`）。

## 第四部分：底层命令实战速查（Plumbing Commands）

如果你想在终端里亲自扒开 Git 的底层看一看，可以使用以下命令：

| Git 核心命令 | 实际作用 |
| :--- | :--- |
| `git cat-file -t <哈希值>` | 探测对象类型：查看该哈希属于 `blob`、`tree` 还是 `commit` |
| `git cat-file -p <哈希值>` | 打印对象内容：查看具体的文本代码，或是树对象的目录列表 |
| `git hash-object -w <文件名>`| 核心算法体验：手动计算一个文件的哈希值，并将其存入 Git 数据库 |
| `git rev-parse HEAD` | 解析指针：查看当前工作区所指向的最顶层 Commit 的完整哈希值 |
